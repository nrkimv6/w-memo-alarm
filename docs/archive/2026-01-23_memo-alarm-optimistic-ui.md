## 개요

메모 추가 시 서버 응답을 기다리지 않고 **즉시 UI에 반영**하는 Optimistic UI 패턴 적용.
실패 시 사용자에게 명확히 알리고 백그라운드 재시도 수행.

### 현재 문제점

```
[현재 흐름 - 로그인 시]
UI 입력 → Supabase INSERT (대기) → 응답 → Realtime → UI 업데이트
         ↑                                        ↑
         네트워크 지연                            여기서야 보임
```

- 서버 응답까지 UI가 멈춤 (사용자가 추가된 메모를 못 봄)
- 네트워크 실패 시 아무것도 표시 안됨
- 재시도 메커니즘 없음

### 목표 흐름

```
[Optimistic UI 흐름]
UI 입력 → 즉시 로컬 저장 + UI 반영 → 백그라운드 동기화
                ↓                           ↓
           바로 보임              성공: 동기화 완료 표시
                                  실패: 토스트 + 동기화 아이콘 + 재시도
```

## 기술적 고려사항

### 1. 메모 상태 확장

```typescript
interface Memo {
  // 기존 필드...
  syncStatus?: 'pending' | 'synced' | 'failed';  // 동기화 상태
  localId?: string;    // 로컬에서 생성한 임시 ID
  retryCount?: number; // 재시도 횟수
}
```

### 2. 동기화 큐 구조

```typescript
interface SyncQueue {
  pending: Memo[];      // 동기화 대기 중
  failed: Memo[];       // 실패한 메모 (재시도 대상)
  isProcessing: boolean;
}
```

### 3. 재시도 전략

- 지수 백오프: 1초 → 2초 → 4초 → 8초 → 16초 (최대)
- 최대 재시도: 5회
- 5회 실패 시: 사용자에게 수동 재시도 버튼 제공

### 4. ID 충돌 방지

- 로컬 ID: `local_${timestamp}_${random}` 형식
- 서버 동기화 성공 시 서버 ID로 교체
- Realtime 이벤트에서 중복 방지 로직 필요

### 5. UI 표시

```
[메모 카드]
┌─────────────────────────────┐
│ 🔄 회의 메모                 │  ← 동기화 중 (pending)
│ 내용...                     │
└─────────────────────────────┘

┌─────────────────────────────┐
│ ⚠️ 회의 메모          [재시도]│  ← 동기화 실패 (failed)
│ 내용...                     │
└─────────────────────────────┘

┌─────────────────────────────┐
│ ✓ 회의 메모                  │  ← 동기화 완료 (synced) - 아이콘 숨김
│ 내용...                     │
└─────────────────────────────┘
```

## 테스트 시나리오

1. **정상 케이스**: 메모 추가 → 즉시 표시 → 1-2초 후 동기화 완료
2. **네트워크 지연**: 메모 추가 → 즉시 표시 + 🔄 아이콘 → 동기화 완료 후 아이콘 숨김
3. **네트워크 실패**: 메모 추가 → 즉시 표시 → 실패 토스트 + ⚠️ 아이콘 → 재시도 버튼
4. **오프라인**: 메모 추가 → 즉시 표시 → 온라인 복구 시 자동 동기화
