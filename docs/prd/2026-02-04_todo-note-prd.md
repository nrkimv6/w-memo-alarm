# 할일(Todo) 메모 타입 PRD

> **상태**: 📋 기획 중
> **작성일**: 2026-02-04
> **우선순위**: P0 (핵심 기능)

---

## 1. 개요

### 1.1 목적

기존 메모(note)와 북마크(bookmark) 타입에 이어 **할일(todo)** 타입을 추가한다.
단순한 체크리스트가 아니라, **반복 할일**, **시간 기반 알림 3단계**, **미루기/건너뛰기** 등
실생활에서 필요한 할일 관리의 핵심 기능을 제공한다.

### 1.2 핵심 콘셉트

```
┌─────────────────────────────────────────────────┐
│                📝 메모 (Memo)                    │
│  ┌───────────┬───────────┬───────────┐          │
│  │  📄 노트   │  🔗 북마크  │  ✅ 할일   │          │
│  │  (note)   │ (bookmark)│  (todo)   │          │
│  └───────────┴───────────┴───────────┘          │
│                                                  │
│  ✅ 할일 (Todo)                                  │
│  ┌───────────────────────────────────────────┐  │
│  │ 제목: 우유 사기                             │  │
│  │ 상태: ⬜ 미완료                             │  │
│  │ 기한: 2026-02-05 18:00                     │  │
│  │ 알림: 상기 → 알람 → 기한초과                 │  │
│  │ 반복: 매주 수요일                            │  │
│  │ 우선순위: 보통                               │  │
│  └───────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 1.3 기존 기능과의 차이

| 구분 | 기본 알람 (Reminder) | 할일 (Todo) |
|------|---------------------|-------------|
| **목적** | 특정 시간에 알려줌 | 해야 할 일을 추적 |
| **완료 개념** | 없음 (알림만) | 있음 (done/미완료) |
| **반복** | 시간 반복 (매일 9시) | 태스크 반복 (완료 후 다음 인스턴스 생성) |
| **기한** | 없음 | 있음 (deadline) |
| **미루기** | 없음 | 있음 (postpone/skip) |
| **상태 추적** | 발송 성공/실패 | 미완료 → 완료/건너뜀/미룸 |

## 2. 사용자 시나리오

### 시나리오 1: 단발성 할일

> 사용자가 "내일까지 보고서 제출"이라는 할일을 만든다.
> 당일 오전에 상기 알림을 받고, 오후 3시에 "지금 해" 알람을 받는다.
> 완료 후 체크하면 할일이 done 처리된다.

### 시나리오 2: 반복 할일

> 사용자가 "매주 수요일 우유 사기"라는 반복 할일을 만든다.
> 매주 수요일마다 새 할일 인스턴스가 생성된다.
> 장기 출장 시 해당 주를 "건너뛰기" 처리한다.
> 수요일에 못 사면 "미루기"로 목요일로 옮긴다.

### 시나리오 3: 할일 모아보기

> 사용자가 "할일" 탭에서 오늘/이번주 할일을 한눈에 본다.
> 미완료 할일은 상단, 완료된 건 하단에 표시된다.
> 기한 초과 할일은 빨간색으로 강조된다.

### 시나리오 4: 그룹 할일

> "출장 준비"라는 그룹 아래 "항공권 예약", "호텔 예약", "짐 싸기"를 묶는다.
> 그룹 진행률(2/3 완료)이 표시된다.

## 3. 핵심 개념 설계

### 3.1 알림 3단계 (시간 기반 상태 전이)

할일의 가장 핵심적인 차별점이다. 기존의 단순 알림(시간 → 알림)이 아니라,
할일의 생명주기에 따라 **3단계로 알림 강도가 달라진다**.

```
시간 흐름 →

[상기 시점]          [알람 시점]          [기한 시점]
    │                    │                    │
    ▼                    ▼                    ▼
 ┌──────┐           ┌──────┐           ┌──────────┐
 │ 상기  │    →      │ 알람  │    →      │ 기한 초과 │
 │Remind │           │Alert │           │ Overdue  │
 └──────┘           └──────┘           └──────────┘
  일반 푸시         발등에 불!            놓침 표시
 노티피케이션     화면 전체/고정노티      빨간 배지
 "잊지 마세요"    "지금 하세요!"       "기한 지남!"
```

#### 3.1.1 각 단계 상세 정의

**1단계: 상기 (Remind) — 노티피케이션**

- **성격**: 가벼운 리마인더. "이런 게 있어" 수준.
- **트리거**: 각 할일의 개별 상기 시각 (remindTimes)
- **동작**: 일반 푸시 알림 (OS 알림 트레이에 표시)
- **내용 예시**: "📋 상기: 우유 사기 (기한: 오늘 18:00)" — 할일별 개별 발송
- **기한 없는 할일도 포함**: 기한이 설정되지 않은 할일도 상기 알림에 포함됨
- **상기 범위**: 상기일자가 오늘인 것 → 각 할일의 개별 상기 시각에 발송
- **설정**: todo별 개별설정(0~N개). 앱 공통설정에서 기본 상기 시각을 설정하면 useGlobalRemind=true인 할일에 일괄 적용
- **차후 개선 아이디어**: 기한이 오늘인 할일을 공통상기시간에 요약 발송하는 기능 (공통상기 방식 추가 검토 필요)

**2단계: 알람 (Alert) — 발등의 불**

- **성격**: "사용자가 하겠다고 마음먹은 시점". 지금 당장 해야 할 때.
- **트리거**: 사용자가 직접 지정한 알람 시각
- **동작**: 강조 알람 (진동/소리 + 가능한 경우 화면 전체 표시 or 고정 노티피케이션)
- **내용 예시**: "🔔 지금! 우유 사러 가세요"
- **설정**: 자동알람시간(기한 N시간 전) 공통설정(on/off) + todo별 개별설정(0~N개)
- **기한 없는 할일**: 자동알람 불가 (수동으로 알람 시각을 설정해야 함)

**3단계: 기한초과 (Overdue) — 기준시각 경과**

- **성격**: 기한 = 기준 시각. 이 시각이 지나면 "놓쳤다"는 상태.
- **트리거**: `dueDate + dueTime < now` (연산된 상태, 별도 알림 발송 아님)
- **동작**: 앱 내 빨간 배지 + 미완료 목록 상단 고정
- **기한 없는 할일**: overdue 상태 불가 (기한이 없으므로)

#### 3.1.2 설정 레이어 (공통 vs 개별)

기존 앱의 `defaultReminder` 패턴(공통 설정 → isDefault 플래그로 연동)을 확장한다.

```
┌─────────────────────────────────────────────────────┐
│              앱 설정 (전역 기본값)                      │
│                                                      │
│  상기 (Remind):                                      │
│    ☑ 활성화                                          │
│    시각: 07:00 (매일 이 시각에 할일 요약 알림)          │
│                                                      │
│  자동 알람 (Auto Alert):                             │
│    ☐ 비활성화 (기본: off)                             │
│    기한 전: [하루 전 ▼]                               │
│                                                      │
│  기한초과 표시: ☑ 항상 표시 (기본: on)                 │
│                                                      │
└──────────────────────┬──────────────────────────────┘
                       │
           상속 (todo별 미설정 시 공통값 사용)
                       │
┌──────────────────────▼──────────────────────────────┐
│              Todo별 개별 설정 (0~N개 추가 가능)        │
│                                                      │
│  "우유 사기" 할일:                                    │
│    상기: 앱 기본값 사용 (07:00)                        │
│    알람: + [수요일 10:00 추가]                         │
│          + [수요일 17:00 추가]                         │
│    기한: 수요일 18:00                                 │
│                                                      │
│  "보고서 제출" 할일:                                   │
│    상기: 앱 기본값 사용 (07:00) + [기한 3일 전 추가]    │
│    알람: [월요일 09:00]                               │
│    기한: 화요일 17:00                                 │
└─────────────────────────────────────────────────────┘
```

**상기 알림 예시 (개별 발송):**

```
각 할일의 상기 시각에 개별 푸시 알림이 발송된다.

07:00  [상기] "📋 상기: 우유 사기 (기한: 오늘 18:00)"     ← useGlobalRemind=true
07:00  [상기] "📋 상기: 보고서 제출 (기한: 내일 17:00)"    ← useGlobalRemind=true
07:00  [상기] "📋 상기: 영어 공부 (기한 없음)"              ← useGlobalRemind=true
09:00  [상기] "📋 상기: 회의 준비 (기한: 오늘 14:00)"      ← 개별 상기 시각

※ 차후 개선: 공통상기시간에 요약 형태로 모아 보내는 기능 검토 예정
```

#### 3.1.3 3단계 알림의 시간축 예시

```
"우유 사기" - 기한: 수요일 18:00, 반복: 매주 수

월요일 07:00  [상기] "📋 상기: 우유 사기 (기한: 수 18:00)"      ← 개별 발송
화요일 07:00  [상기] "📋 상기: 우유 사기 (기한: 내일 18:00)"   ← 개별 발송
수요일 07:00  [상기] "📋 상기: 우유 사기 (기한: 오늘 18:00)"   ← 기한 당일
수요일 10:00  [알람] "🔔 지금! 우유 사러 가세요"              ← 사용자 설정 시각
수요일 18:00  [기한] ⚠️ 기한 초과 (앱 내 빨간 표시)           ← 기준시각 경과
수요일 18:01  → overdue 상태 (목록 상단 고정)

"보고서 제출" - 기한: 화요일 17:00, 단발성

금요일 07:00  [상기] "📋 상기: 보고서 제출 (기한: 화 17:00)"     ← 개별 발송
월요일 07:00  [상기] "📋 상기: 보고서 제출 (기한: 내일 17:00)"  ← 기한 하루 전
화요일 07:00  [상기] "📋 상기: 보고서 제출 (기한: 오늘 17:00)"  ← 기한 당일
화요일 09:00  [알람] "🔔 보고서 제출 마감 8시간 전!"            ← 자동알람 (기한 전)
화요일 17:00  [기한] ⚠️ 기한 초과                              ← 기준시각

"영어 공부" - 기한 없음, 반복 없음

매일 07:00   [상기] "📋 상기: 영어 공부 (기한 없음)"              ← 개별 발송
             → 알람(Alert) 없음 (기한 없으므로 자동알람 불가, 수동 설정 가능)
             → 기한초과 없음
```

### 3.2 반복 할일 (Recurring Todo)

**기존 "반복 알람"과의 핵심 차이:**

```
반복 알람 (기존):
  매일 9시 알림 → 매일 9시에 알림이 울림 (완료 개념 없음)

반복 할일 (신규):
  매주 수요일 우유 사기 → 이번 주 수요일 인스턴스 생성
                       → 완료/건너뛰기/미루기 처리
                       → 다음 주 수요일 인스턴스 자동 생성
```

**계층 구조:**

```
┌─ RecurringTodo (반복 할일 템플릿) ──────────────────┐
│  제목: "우유 사기"                                    │
│  반복: 매주 수요일                                    │
│  알람: 수요일 오전 10시                               │
│                                                      │
│  ┌─ TodoInstance (2/5주)  ──┐                        │
│  │  상태: ✅ 완료           │                        │
│  │  완료일: 2/5 11:30       │                        │
│  └──────────────────────────┘                        │
│  ┌─ TodoInstance (2/12주) ──┐                        │
│  │  상태: ⏭️ 건너뜀         │                        │
│  │  사유: 출장              │                        │
│  └──────────────────────────┘                        │
│  ┌─ TodoInstance (2/19주) ──┐                        │
│  │  상태: ⬜ 미완료         │ ← 현재               │
│  └──────────────────────────┘                        │
└──────────────────────────────────────────────────────┘
```

> **내 의견 — "반복할일"을 상위 개념으로 둘 것인가?**
>
> 유저의 질문: "할일의 상위에 반복할일이라는 개념을 두는게 나을까?"
>
> **권장: 같은 레벨, 속성으로 구분한다.**
> - 반복할일을 별도 엔티티로 만들면 데이터 모델이 복잡해지고, 기존 Memo 테이블과의 통합이 어렵다.
> - 대신 Todo 메모에 `recurrence` 속성을 추가하고, 인스턴스는 `todoInstances` 배열로 관리한다.
> - 반복 할일은 "메모 1개 + 인스턴스 N개"의 구조가 된다.
> - UI에서 반복 아이콘(🔁)으로 시각적 구분만 하면 충분하다.

### 3.3 미루기 / 건너뛰기

| 액션 | 동작 | 사용 예 |
|------|------|---------|
| **미루기 (Postpone)** | 기한을 뒤로 이동 | "오늘 못 하겠으니 내일로" |
| **건너뛰기 (Skip)** | 이번 인스턴스 패스 (반복 할일 전용) | "출장이라 이번 주는 안 함" |

```
미루기:
  기한: 2/5(수) → 미루기(+1일) → 기한: 2/6(목)
  - 할일 자체는 유지, 기한만 변경
  - 미룬 횟수 기록 (postponeCount)
  - autoAlertBefore(자동알람) → 새 기한 기준으로 자동 재계산
  - 수동 alertTimes → 이동하지 않음 (사용자가 직접 지정한 시각은 유지. 과거 시각이 되면 무시)
  - 상기(remind) → 새 기한 기준으로 재계산

건너뛰기 (반복 할일 전용):
  2/5(수) 인스턴스 → 건너뜀 처리 → 2/12(수) 인스턴스 자동 생성
  - 해당 인스턴스는 "skipped" 상태로 기록
  - 다음 반복으로 넘어감
```

#### 3.3.1 미루기 에스컬레이션 (2회차부터 경고)

**1회 미루기**: 바로 처리. 빠른 프리셋 표시.

**2회째 미루기부터**: 경고 표시 + 미루기 횟수 제한 토글 (선택적).

```
1회 미루기 (일반):
┌─────────────────────────────────────────┐
│  📌 미루기 - "우유 사기"                 │
├─────────────────────────────────────────┤
│                                         │
│  빠른 선택:                             │
│  [내일] [모레] [이번주말] [다음주]        │
│                                         │
│  날짜 직접 선택: [________]             │
│                                         │
│  [취소]              [미루기]            │
└─────────────────────────────────────────┘

2회째 미루기 (경고 추가):
┌─────────────────────────────────────────┐
│  ⚠️ 미루기 - "우유 사기"                 │
├─────────────────────────────────────────┤
│                                         │
│  이 할일은 이미 1회 미뤄졌습니다.         │
│  (원래 기한: 2/5 → 현재: 2/6)           │
│                                         │
│  미루기 날짜: [내일 ▼]                   │
│                                         │
│  ☐ 미루기 횟수 제한 설정                  │  ← 토글 (기본: off)
│  ┌─────────────────────────────────┐   │
│  │ 앞으로 최대: [2회 ▼] 까지 허용   │   │  ← 토글 on 시 표시
│  │   ├ 1회                         │   │
│  │   ├ 2회                         │   │
│  │   ├ 3회                         │   │
│  │   └ 5회                         │   │
│  │                                 │   │
│  │ ⚠️ 소진 시 완료/건너뛰기만 가능  │   │
│  └─────────────────────────────────┘   │
│                                         │
│  [취소]              [미루기]            │
└─────────────────────────────────────────┘
```

**maxAllowed는 설정 항목이 아니다.**
미루기 시점에 사용자가 선택적으로 입력하는 값이며, 기본값은 `null` (제한 없음).
토글을 켜야만 횟수 제한이 활성화된다.

**미루기 제한 로직:**

```typescript
interface PostponeInfo {
  count: number;              // 현재까지 미룬 횟수
  maxAllowed: number | null;  // null = 제한없음 (기본). 2회차에 사용자가 토글 on하면 설정
  originalDueDate: string;    // 최초 기한 (미루기 전)
  history: PostponeRecord[];  // 미루기 이력
}

interface PostponeRecord {
  from: string;               // 이전 기한
  to: string;                 // 새 기한
  postponedAt: string;        // 미룬 시각
}

// 미루기 가능 여부 판단
function canPostpone(info: PostponeInfo): boolean {
  if (info.maxAllowed === null) return true;  // 제한 없음
  return info.count < info.maxAllowed;
}
```

**미루기/건너뛰기 동작 기준: "다음 반복 일정" 기준**

반복 할일에서 미루기/건너뛰기는 모두 **다음 반복 일정(알람 시각) 기준**으로 동작한다.

```
"매주 수요일 우유 사기" — 기한: 수요일 18:00

현재: 2/5(수) 인스턴스
  → 미루기: 2/6(목)으로 이동. 이 인스턴스의 기한만 변경.
            2/12(수) 다음 인스턴스는 영향 없음.
  → 건너뛰기: 2/5 인스턴스 skip → 즉시 2/12(수) 인스턴스 생성.

미룬 후 또 미루기 (2회차):
  기한: 2/6(목) → 2/7(금)으로 이동.
  경고: "이미 1회 미뤘습니다."
  사용자가 토글 on + "2회까지" 입력 → maxAllowed = 2 (앞으로 2회 = 총 3회)

  3회째 미루기: OK (아직 maxAllowed 이내)
  4회째 미루기 시도:
  → maxAllowed 초과 → 미루기 버튼 비활성화
  → "완료" 또는 "건너뛰기"만 가능

  토글 off한 경우: maxAllowed = null → 무제한 미루기 가능
  (2회째부터 매번 경고는 표시되지만 제한은 없음)
```

### 3.4 할일 상태 머신

```
                    ┌──────────┐
          생성 ──→  │  pending  │
                    └──────────┘
                         │
              ┌──────────┼──────────────┐
              ▼          ▼              ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │completed │ │postponed │ │ skipped  │
        │  (완료)   │ │ (미뤄짐)  │ │(건너뜀)  │
        └──────────┘ └──────────┘ └──────────┘
                         │
                         ▼
                    ┌──────────┐
                    │  pending  │  (새 기한으로 재생성)
                    └──────────┘
```

**기한 초과 시:**

```
pending + (현재시각 > dueDate) → overdue (기한초과)
  │
  ├─→ 완료 처리 가능 (늦었지만 함)
  ├─→ 미루기 가능 (기한 재설정)
  └─→ 건너뛰기 가능 (반복 할일)
```

> `overdue`는 별도 상태값이 아니라 **연산된 상태**로 처리한다. (pending + dueDate < now = overdue)

```typescript
// 기존 Memo 인터페이스에 추가되는 할일 관련 필드

interface Memo {
  // ... 기존 필드 유지 ...
  memoType?: 'note' | 'bookmark' | 'todo';   // 'todo' 신규 추가 (기존에 'task' 사용 이력 없음)

  // === 할일 속성 (memoType === 'todo'일 때 사용) ===
  todoStatus?: TodoStatus;
  todoPriority?: TodoPriority;
  dueDate?: string;           // YYYY-MM-DD (기존 필드 활용)
  dueTime?: string;           // HH:mm

  // 알림 3단계
  todoTiming?: TodoTiming;

  // 반복 할일
  recurrence?: Recurrence;
  todoInstances?: TodoInstance[];

  // 미루기 추적
  postponeInfo?: PostponeInfo;

  // 그룹
  todoGroupId?: string;       // 할일 그룹 소속
}

interface PostponeInfo {
  count: number;                // 현재까지 미룬 횟수
  maxAllowed: number | null;    // null = 제한없음 (기본). 2회차 미루기 시 사용자가 토글 on하면 설정
  originalDueDate?: string;     // 최초 기한 (미루기 전)
  history: PostponeRecord[];    // 미루기 이력
}

interface PostponeRecord {
  from: string;                 // 이전 기한
  to: string;                   // 새 기한
  postponedAt: string;          // 미룬 시각 (ISO)
}

type TodoStatus = 'pending' | 'completed' | 'skipped';
type TodoPriority = 'low' | 'medium' | 'high' | 'urgent';

interface TodoTiming {
  // 상기 (Remind) — 개별 설정. 미설정 시 앱 공통 상기시간 사용.
  remindTimes?: TodoRemindEntry[];   // 0~N개 (추가 상기 시각)
  useGlobalRemind?: boolean;         // true = 앱 공통설정 사용 (기본: true)

  // 알람 (Alert) — 개별 설정
  alertTimes?: TodoAlertEntry[];     // 0~N개 (직접 지정 알람 시각)
  autoAlertBefore?: number | null;   // 자동알람: 기한 N분 전 (null=사용안함)
  useGlobalAutoAlert?: boolean;      // true = 앱 공통설정 사용 (기본: true)
}

interface TodoRemindEntry {
  id: string;
  type: 'time' | 'before_due';       // 'time'=매일 특정시각, 'before_due'=기한 N분전
  time?: string;                      // HH:mm (type='time')
  minutesBefore?: number;             // 기한 N분 전 (type='before_due')
}

interface TodoAlertEntry {
  id: string;
  time: string;                       // HH:mm - 알람 시각
  date?: string;                      // YYYY-MM-DD (단발성) or 없으면 기한일 사용
  days?: number[];                    // 요일 (반복할일용)
}

interface Recurrence {
  type: 'daily' | 'weekly' | 'monthly' | 'custom';
  interval: number;           // 반복 간격 (매 N일/주/월)
  daysOfWeek?: number[];      // 요일 [0-6] (weekly용)
  dayOfMonth?: number;        // 월 중 일자 (monthly용)
  endDate?: string;           // 반복 종료일 (없으면 무한)
  endAfter?: number;          // N회 후 종료
}

interface TodoInstance {
  id: string;
  scheduledDate: string;      // YYYY-MM-DD 예정일
  status: TodoStatus;
  completedAt?: string;       // 완료 시각
  skippedAt?: string;         // 건너뛴 시각
  skipReason?: string;        // 건너뛰기 사유 (선택)
  postponedTo?: string;       // 미뤄진 날짜
  postponeCount: number;
}
```

## 5. UI/UX 설계

### 5.1 할일 표시 방식 결정

유저의 질문: "탭(하단 네비바에 추가) vs 하나의 메모처럼 표시?"

> **내 의견: 둘 다 한다. (하이브리드 접근)**
>
> 1. **메모 목록에서는 하나의 카드로 표시** — 기존 메모 흐름 유지
> 2. **하단 네비바에 "할일" 탭 추가** — 할일만 모아보는 전용 뷰
>
> 이유:
> - 할일만 빠르게 보고 싶을 때가 있고 (→ 전용 탭)
> - 메모/북마크와 섞여 있는 맥락에서 보고 싶을 때도 있다 (→ 메모 목록)
> - 기존 4탭(홈/전체메모/알림내역/설정)에 "할일" 1탭 추가는 5탭이 되어 네비바가 좁아질 수 있다
> - **대안: 홈 화면에 "오늘의 할일" 섹션 추가 + /todos 라우트를 상단 탭이나 필터로 접근**

**결정된 네비게이션 구조 (4탭):**

```
하단 네비바 (4탭):
[홈 🏠] [메모 📋] [할일 ✅] [설정 ⚙️]

- 알림내역은 홈 상단 벨 아이콘 또는 설정 내 하위메뉴로 접근
- 홈에 "오늘의 할일" 섹션 표시
```

### 6.3 설정 ↔ Todo 연동 로직

기존 `defaultReminder` 패턴을 따른다:

```
사용자가 설정 변경
  → settingsStore.setTodoDefaults(newSettings)

    → memosStore.updateGlobalTodoSettings()

    → notificationStore.syncTodoRemindersToServiceWorker()
      → SW에 할일 상기/알람 스케줄 갱신
```

### 6.4 기본값 정리

| **기한초과** | 해당 없음 |
| **수동 알람** | 사용자가 직접 시각 지정 가능 |

## 7. 알림 시스템 통합

### 7.1 할일 알림 vs 기존 알림 구분

```
기존 시스템:
  Memo.reminders[] → 시간 기반 반복 알림
  "매일 9시에 알려줘"

할일 시스템 (신규):
  Memo.todoTiming → 할일 기한 연동 알림
  "기한 하루 전에 상기, 당일 오전에 알람, 기한 지나면 오버듀"
```

**공존 방식:**
- 할일(todo) 타입 메모에서도 기존 `reminders[]`를 사용할 수 있다 (추가 알림)
- `todoTiming`은 기한(dueDate) 기반으로 자동 계산되는 알람
- 두 시스템은 독립적으로 동작하되, 알림내역(NotificationHistory)에 함께 기록

### 7.2 알림 채널별 동작

| 채널 | 상기 (Remind) | 알람 (Alert) | 기한초과 (Overdue) |
|------|--------------|-------------|-------------------|
| **Service Worker** | 일반 푸시 | 일반 푸시 | - (앱 내 표시) |
| **Capacitor** | 일반 알림 | 진동+소리 알람 | - (앱 내 표시) |
| **FCM** | 일반 푸시 | 우선순위 높은 푸시 | - (앱 내 표시) |
| **앱 내** | - | - | 배지 + 목록 상단 고정 |

### 7.3 반복 할일의 인스턴스 생성 시점

**결정: 직전 생성 (Just-in-Time) + 미래 일정은 계산으로 표시**

DB/localStorage에 저장되는 TodoInstance는 **과거 기록 + 활성(pending) 인스턴스**이다.
일반적으로 활성 인스턴스는 1개이지만, overdue 인스턴스가 처리되지 않은 채 다음 정규 일정이 도래하면 **활성 인스턴스가 2개 이상** 쌓일 수 있다.
모아보기에서 미래 일정을 보여줄 때는 `recurrence` 패턴에서 on-the-fly로 계산한다.

```
인스턴스 생성 시점:

1. 반복 할일 최초 생성 시
   → 첫 번째 인스턴스 1개만 생성 (가장 가까운 다음 일정)
   → DB에 저장: todoInstances = [{ scheduledDate: '2/5', status: 'pending' }]

2. 현재 인스턴스 완료(completed) 시
   → 현재 인스턴스 상태를 'completed'로 변경
   → 다음 인스턴스 1개 자동 생성 (recurrence 패턴 기반)
   → DB: todoInstances = [...과거, { scheduledDate: '2/12', status: 'pending' }]

3. 현재 인스턴스 건너뛰기(skipped) 시
   → 현재 인스턴스 상태를 'skipped'로 변경
   → 다음 인스턴스 1개 자동 생성
   → DB: todoInstances = [...과거, { scheduledDate: '2/12', status: 'pending' }]

4. 미루기(postpone) 시
   → 새 인스턴스 생성하지 않음
   → 현재 인스턴스의 scheduledDate만 변경 (예: 2/5 → 2/6)
   → 다음 정규 일정(2/12)은 영향 없음

5. 앱 시작 시 (누락 복구)
   → 현재 활성 인스턴스(status='pending')가 없고
     마지막 인스턴스가 completed/skipped인 경우
   → 다음 인스턴스 자동 생성
   → (오프라인 장기 사용 후 앱 열 때 필요)

6. 앱 시작 시 (overdue 누적)
   → 활성 인스턴스가 overdue이고, 다음 정규 일정이 이미 도래한 경우
   → 다음 인스턴스도 생성 (overdue 인스턴스 + 새 인스턴스 공존)
   → 자동 skip 안 함 — overdue는 사용자가 직접 완료/건너뛰기/미루기 처리
   → 예: 매주 수 반복, 2/5 overdue 미처리 → 2/12 도래 → 2/5 + 2/12 두 개 활성
```

**일정별 통합 조회 시 데이터 소스 구분:**

```
┌──────────────────────────────────────────────────┐
│  할일 모아보기에서 보여주는 항목들                     │
├──────────────────────────────────────────────────┤
│                                                  │
│  실제 데이터 (DB 저장)          계산된 데이터 (메모리)  │
│  ─────────────────          ──────────────────  │
│                                                  │
│  • 단발성 할일                • 반복 할일의 미래 일정   │
│    (Memo 자체가 할일)          (recurrence에서 계산)  │
│                                                  │
│  • 반복 할일의 현재 인스턴스    • "2주 후 우유사기"     │
│    (TodoInstance, pending)    같은 미래 표시는       │
│                              getUpcomingSchedules()│
│  • 반복 할일의 과거 인스턴스    로 on-the-fly 계산    │
│    (completed/skipped)                            │
│                                                  │
├──────────────────────────────────────────────────┤
│  미루기/건너뛰기 대상:                               │
│  → 현재 활성 인스턴스(pending)만 대상                 │
│  → 미래 계산 일정에는 미루기/건너뛰기 불가             │
│    (아직 생성되지 않은 인스턴스이므로)                  │
└──────────────────────────────────────────────────┘
```

**왜 미래 인스턴스를 미리 생성하지 않는가:**

1. 미루기/건너뛰기는 현재 인스턴스에만 적용됨 — 미래 것은 아직 존재할 필요 없음
2. 반복 패턴이 변경되면 (매주→매일) 미리 생성한 것을 전부 삭제/재생성해야 함
3. 무한 반복 할일의 경우 생성할 인스턴스 수를 정할 수 없음
4. 모아보기에서 미래 일정을 보여주는 건 계산으로 충분함

## 8. 기술적 한계 분석 — Alert 화면 표시

### 8.1 플랫폼별 기능 매트릭스

할일의 2단계 알람(Alert)에서 "알람시계처럼 화면 전체 표시" 또는 "고정 노티피케이션"이
가능한지 플랫폼별로 분석한다.

| 기능 | PWA (웹) | Native (Capacitor) | 비고 |
|------|---------|-------------------|------|
| **화면 전체 그리기 (Fullscreen Alert)** | ❌ 불가 | ✅ 가능 | 아래 상세 |
| **고정 노티피케이션 (Persistent/Ongoing)** | ⚠️ 제한적 | ✅ 가능 | 아래 상세 |
| **진동** | ⚠️ 제한적 | ✅ 가능 | |
| **소리 (커스텀)** | ⚠️ 제한적 | ✅ 가능 | |
| **잠금화면 표시** | ❌ 불가 | ✅ 가능 | |

### 8.2 PWA 상세 한계

**화면 전체 그리기 (Fullscreen Alert) — ❌ 불가**

- 웹 브라우저 보안 정책상, 백그라운드에서 **새 창을 열거나 전체화면을 강제**할 수 없음
- `Fullscreen API`는 사용자 제스처(클릭/키 입력)에 의해서만 동작
- Service Worker에서 `clients.openWindow()`로 새 탭을 열 수는 있으나:
  - 사용자가 알림을 클릭해야만 실행됨 (자동 불가)
  - 전체화면이 아닌 일반 브라우저 탭으로 열림
- **대안**: 알림 클릭 시 할일 상세 페이지를 열어서 "완료/미루기" 액션 유도

**고정 노티피케이션 (Persistent) — ⚠️ 제한적**

- Web Notification API의 `requireInteraction: true` 옵션으로 자동 닫힘 방지 가능
  - 현재 앱에서 이미 사용 중 (`src/service-worker.ts`)
  - 사용자가 직접 닫거나 클릭할 때까지 유지
- 단, **OS별 동작이 다름**:
  - Windows: requireInteraction 지원됨 (닫을 때까지 유지)
  - macOS: 일부 브라우저에서 무시됨 (자동 닫힘)
  - Android Chrome: 지원됨
  - iOS Safari: Web Push 자체가 매우 제한적
- **Ongoing(진행 중) 노티피케이션**: 불가. 웹에서는 "상태바 고정" 타입 알림 생성 불가.

**진동 — ⚠️ 제한적**

- `Vibration API`는 포그라운드에서만 동작
- Service Worker에서 발송하는 푸시 알림에는 `vibrate` 패턴 설정 가능하지만,
  실제 동작은 OS 설정에 의존

**소리 — ⚠️ 제한적**

- Web Notification에 `sound` 속성은 표준이 아님 (Chrome에서 무시)
- 포그라운드에서 `Audio API`로 재생은 가능하나 백그라운드에서는 불가

### 8.3 Native App (Capacitor) 상세

**화면 전체 그리기 (Fullscreen Alert) — ✅ 가능**

- Android: `FLAG_SHOW_WHEN_LOCKED`, `FLAG_TURN_SCREEN_ON`, `FLAG_KEEP_SCREEN_ON`
  - Activity를 FULL_SCREEN_INTENT로 설정하면 알람시계처럼 화면 전체 표시 가능
  - Capacitor 커스텀 플러그인 또는 네이티브 코드(Java/Kotlin)로 구현
  - `@capacitor/local-notifications`의 `schedule()`에 `extra` 필드로 fullscreen 플래그 전달 가능
- iOS: `UNNotificationContentExtension`으로 커스텀 알림 UI 가능 (전체화면은 아님)

**구현 방법 (Android):**

```java
// Android 네이티브 코드 (Capacitor 플러그인)
public class FullscreenAlarmActivity extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);

        // 잠금화면 위에 표시
        setShowWhenLocked(true);
        setTurnScreenOn(true);

        // 키가드 해제
        KeyguardManager km = getSystemService(KeyguardManager.class);
        km.requestDismissKeyguard(this, null);

        // 전체화면 UI 표시
        setContentView(R.layout.activity_fullscreen_alarm);
    }
}
```

**고정 노티피케이션 (Ongoing) — ✅ 가능**

- Android: `setOngoing(true)`로 사용자가 스와이프로 제거 불가한 알림 생성
- Capacitor 커스텀 플러그인으로 구현 필요 (`@capacitor/local-notifications` 기본 지원은 없음)
- 할일 기한까지 상태바에 상주하며 남은 시간 표시 가능

**진동 / 소리 — ✅ 가능**

- 커스텀 진동 패턴, 커스텀 알림음 모두 지원
- Android Notification Channel로 Alert 전용 채널 생성 (높은 우선순위)

### 8.4 플랫폼별 Alert 구현 전략

| 플랫폼 | Alert 동작 | 구현 |
|--------|-----------|------|
| **PWA (웹)** | `requireInteraction: true` 푸시 + 알림 클릭 시 앱 열기 | 기존 SW 활용 |
| **PWA (포그라운드)** | 앱 내 모달 오버레이 (전체화면 차지) + 소리/진동 | Audio API + 커스텀 UI |
| **Native (백그라운드)** | Fullscreen Intent Activity (전체화면 알람) | Capacitor 커스텀 플러그인 |
| **Native (포그라운드)** | 앱 내 모달 오버레이 (전체화면 차지) + 소리/진동 | 동일 |

> **의견**: PWA에서는 "알람시계 스타일 화면 전체"가 불가하므로,
> PWA에서의 Alert는 `requireInteraction` 푸시 + 앱 내 진입 시 모달 오버레이로 대체한다.
> Native에서만 true fullscreen alert를 구현하되, Capacitor 커스텀 플러그인이 필요하므로 Phase 2로 분리한다.
> Phase 1에서는 PWA/Native 동일하게 "강조 푸시 알림 + 앱 내 모달" 방식으로 통일.

## 9. 할일 모아보기 — 반복 일정 기준 뷰

### 9.1 핵심 개념

할일 모아보기에서 반복 할일은 **다음 반복 일정(알람 시각) 기준**으로 표시한다.
미루기/건너뛰기도 이 일정 기준으로 동작한다.

```
"매주 수요일 우유 사기" 반복 할일의 다음 일정들:

현재 시각: 2/4(화) 15:00

다음 일정 기준으로 표시:
  2/5(수) 18:00  ← 이번 주 (다음 인스턴스, 활성)
  2/12(수) 18:00 ← 다음 주 (미리보기)
  2/19(수) 18:00 ← 다다음 주 (미리보기)
  ...

할일 모아보기에서는 가장 가까운 인스턴스(2/5)가 표시됨.
"이번주" 탭에서는 2/5만, "전체" 탭에서는 미래 일정도 연하게 표시.
```

### 9.3 반복 할일의 일정 계산

```typescript
/**
 * 반복 할일의 "다음 N개 일정"을 계산하는 함수.
 * 모아보기에서 미래 인스턴스를 미리 보여주기 위해 사용.
 * (실제 TodoInstance 생성은 직전 생성 방식 유지)
 */
function getUpcomingSchedules(
  memo: Memo,
  count: number = 5
): SchedulePreview[] {
  if (!memo.recurrence) return [];

  const schedules: SchedulePreview[] = [];
  let nextDate = getNextOccurrence(memo.recurrence, new Date());

  for (let i = 0; i < count; i++) {
    schedules.push({
      date: nextDate,
      isPostponed: false,
      isActive: i === 0,   // 가장 가까운 것만 활성
    });
    nextDate = getNextOccurrence(memo.recurrence, addDays(nextDate, 1));
  }

  return schedules;
}

interface SchedulePreview {
  date: Date;
  isPostponed: boolean;
  isActive: boolean;       // 현재 활성 인스턴스 여부
}
```

### 9.4 기한 없는 할일의 정렬 위치

기한이 없는 할일은 **"오늘" 섹션의 하단**에 표시한다.

```
정렬 우선순위:
  1. overdue (기한 초과) — 빨간 섹션, 최상단
  2. 오늘 기한 할일 — 시각 순
  3. 오늘 알람 있는 할일 — 알람 시각 순
  4. 기한 없는 미완료 할일 — 생성일 순 ← 여기
  5. 오늘 완료된 할일 — 완료 시각 역순
```

## 10. 추가 고려사항

### 10.1 그룹핑에 대한 의견
>
> **Phase 1**: 기존 `tags`와 `folders`로 그룹핑을 대체
> - 태그로 "장보기", "업무", "집안일" 등 분류
> - 폴더로 큰 카테고리 분류
>
> **Phase 4**: 전용 `TodoGroup` 도입 (필요성 검증 후)
> - 그룹 진행률 표시 (2/5 완료)
> - 그룹 단위 미루기/건너뛰기
> - 그룹 내 할일 순서 지정
>
> 이유: 그룹핑은 유용하지만, 태그/폴더와 기능이 겹칠 수 있다.
> 먼저 태그/폴더로 충분한지 검증 후 전용 그룹을 도입하는 것이 안전하다.

### 10.2 추가로 고려해야 할 사항

**1. 완료 후 되돌리기 (Undo)**
- 실수로 완료 체크했을 때 되돌리기 필요
- 토스트 메시지로 "실행 취소" 버튼 제공 (3초간)

**2. 대량 처리 (Batch Actions)**
- 여러 할일 선택 → 일괄 완료/미루기/삭제
- 특히 기한 지난 할일이 쌓였을 때 필요

**3. 위젯/홈 화면 연동**
- Android 홈 화면 위젯에 오늘의 할일 표시
- Capacitor 플러그인으로 네이티브 위젯 제공 가능 (향후)

**4. 캘린더 뷰**
- 할일을 달력 형태로 보는 뷰
- 기한별 분포 확인, 드래그로 기한 변경
- Phase 2 이후 고려

**5. 완료 통계/스트릭**
- 연속 완료 일수 (스트릭)
- 주간/월간 완료율
- 가장 많이 미룬 할일 Top 3
- 동기부여 요소

**6. 할일 ↔ 메모 전환**
- 일반 메모를 할일로 전환 (memoType 변경)
- 할일 완료 후 메모로 보관 (기록 유지)
- 기존 checklist 가 있는 메모를 할일로 마이그레이션

**7. 알림 피로도 관리**
- 상기+알람+기한초과가 동시에 여러 할일에서 오면 부담
- "방해 금지 시간대" 설정
- 알림 병합 (여러 할일을 하나의 알림으로 묶기)
- 우선순위가 높은 것만 강제 알림, 나머지는 조용한 알림

**8. 오프라인 동작**
- 오프라인에서 할일 완료/미루기 처리
- 온라인 복귀 시 Supabase 동기화
- 기존 optimistic UI 패턴 활용

**9. 자연어 입력 (향후)**
- "매주 수요일 우유 사기" → 자동 파싱하여 반복+제목 설정
- "내일 3시까지 보고서" → 기한 자동 설정
- Phase 2 이후 검토

**10. 서브태스크 vs 체크리스트**
- 기존 `checklist` 필드가 있는데, 이것과 할일의 관계 정리 필요
- 체크리스트: 할일 내부의 세부 항목 (할일 1개에 딸린 리스트)
- 서브태스크: 별도의 할일이 부모 할일에 연결 (1:N 관계)
- Phase 1에서는 기존 체크리스트를 활용하고, 서브태스크는 Phase 2에서 검토

## 11. TBD — 향후 검토 사항

### 11.1 스마트 알림/알람 시각 제안 (배터리 옵티마이저 연동)

> **전제**: 사용자의 배터리 최적화(Doze 모드) 설정이 해제되어 있어야 정확한 알림이 가능.
> 배터리 옵티마이저가 활성화된 경우, 알림이 지연되거나 누락될 수 있음.

**배터리 옵티마이저 미해제 시 대안:**

```
사용자가 배터리 최적화 해제를 거부한 경우:
  → "정확한 알림"이 보장되지 않음
  → 대안: 스마트 알림 시각 산정

스마트 알림 시각이란:
  - Android Doze 모드의 "유지보수 윈도우" 타이밍에 맞춰 알림을 보내는 방식
  - 사용자가 원하는 시각 ± 15분 이내에 실제 알림이 도달하도록 조정
  - AlarmManager.setExactAndAllowWhileIdle() 대신
    setAndAllowWhileIdle()를 사용하되, 최적 윈도우 계산
```

**구현 가능성 분석:**

| 항목 | 가능 여부 | 비고 |
|------|----------|------|
| 배터리 최적화 상태 감지 | ✅ | `PowerManager.isIgnoringBatteryOptimizations()` |
| 배터리 최적화 해제 요청 | ✅ | `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` 인텐트 |
| Exact Alarm 권한 | ✅ | Android 12+ `SCHEDULE_EXACT_ALARM` 권한 |
| Doze 윈도우 예측 | ❌ | OS가 윈도우 시점을 공개하지 않음 |
| 핸드폰 사용 패턴 분석 | ⚠️ | 아래 11.2 참조 |

**권장 구현 (Phase 1):**

```
앱 최초 실행 시:
  → 배터리 최적화 상태 확인
  → 해제되지 않았으면 안내 다이얼로그 표시
  → "정확한 알림을 위해 배터리 최적화를 해제해 주세요"
  → [해제하기] [나중에] [다시 보지 않기]

해제 거부 시:
  → 설정 페이지에 경고 배너 표시
  → "⚠️ 배터리 최적화가 활성화되어 알림이 지연될 수 있습니다"
```

### 11.2 로컬 모델 사용 — 사용자 패턴 분석

> **목표**: 사용자의 핸드폰 사용 패턴을 분석하여 최적의 알림/알람 시각을 제안.

**제안 시나리오:**

```
사용자가 "우유 사기" 할일 생성 (기한: 수요일 18:00)

앱이 분석:
  - 사용자는 보통 수요일 17:30에 퇴근 이동을 시작함 (위치/스크린 패턴)
  - 마트는 보통 퇴근 경로에 있음
  - 17:20에 알람을 보내면 퇴근 전에 준비 가능

앱 제안:
  "📋 추천 알람 시각: 17:20 (퇴근 10분 전)"
  [적용] [다른 시각 선택]
```

**기술 구현 방향:**

```
Phase 1: 단순 통계 기반 (로컬 모델 불필요)
  - 사용자의 알림 클릭 시각 기록
  - 할일 완료 시각 패턴 분석
  - "보통 N시에 이 할일을 완료하시네요" → 알람 시각 추천

  - 핸드폰 사용 시간대 패턴 (스크린 ON/OFF)
  - 앱 사용 패턴 (어떤 시간대에 활동적인지)
  - 과거 할일 완료 패턴 학습
  → 개인화된 최적 알람 시각 산출

Phase 3: 위치 기반 연동 (future-features.md Phase 17 참조)
```

## 14. 요약

| 항목 | 결정 |
|------|------|
| **표시 방식** | 하이브리드 (메모 목록 + 전용 탭) |
| **알림** | 3단계 (상기/알람/기한초과) — 공통설정 + todo별 개별설정 |
| **상기 (Remind)** | 매일 설정시각에 할일 요약 알림 (기한 없는 할일 포함) |
| **알람 (Alert)** | 사용자 지정 시각 + 자동알람(기한 N시간 전) |
| **기한 (Deadline)** | 기준시각. 경과 시 overdue 연산 상태 |
| **반복** | Memo 속성으로 관리 (recurrence + instances) |
| **미루기/건너뛰기** | 둘 다 제공. 2회차부터 에스컬레이션 (횟수 제한 입력) |
| **그룹핑** | Phase 1은 태그/폴더 활용, Phase 4에서 전용 그룹 |
| **Fullscreen Alert (PWA)** | ❌ 불가 → requireInteraction 푸시 + 앱 내 모달 대체 |
| **Fullscreen Alert (Native)** | ✅ 가능 → Phase 4에서 Capacitor 커스텀 플러그인 |
| **고정 노티 (PWA)** | ⚠️ 제한적 → requireInteraction으로 부분 지원 |
| **고정 노티 (Native)** | ✅ 가능 → setOngoing(true) |
| **기존 코드 활용** | dueDate, priority, checklist, defaultReminder 패턴 재활용 |
| **개발 순서** | MVP+설정 → 알림3단계+미루기 → 반복 → Native강화+통계 → 고급 |
