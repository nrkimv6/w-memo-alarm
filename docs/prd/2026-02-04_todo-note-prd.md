## 1. 개요

### 1.1 목적

기존 메모(note)와 북마크(bookmark) 타입에 이어 **할일(todo)** 타입을 추가한다.
단순한 체크리스트가 아니라, **반복 할일**, **시간 기반 알림 3단계**, **미루기/건너뛰기** 등
실생활에서 필요한 할일 관리의 핵심 기능을 제공한다.

### 1.2 핵심 콘셉트

```
┌─────────────────────────────────────────────────┐
│                📝 메모 (Memo)                    │
│  ┌───────────┬───────────┬───────────┐          │
│  │  📄 노트   │  🔗 북마크  │  ✅ 할일   │          │
│  │  (note)   │ (bookmark)│  (todo)   │          │
│  └───────────┴───────────┴───────────┘          │
│                                                  │
│  ✅ 할일 (Todo)                                  │
│  ┌───────────────────────────────────────────┐  │
│  │ 제목: 우유 사기                             │  │
│  │ 상태: ⬜ 미완료                             │  │
│  │ 기한: 2026-02-05 18:00                     │  │
│  │ 알림: 상기 → 알림 → 기한초과                 │  │
│  │ 반복: 매주 수요일                            │  │
│  │ 우선순위: 보통                               │  │
│  └───────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 1.3 기존 기능과의 차이

| 구분 | 기본 알람 (Reminder) | 할일 (Todo) |
|------|---------------------|-------------|
| **목적** | 특정 시간에 알려줌 | 해야 할 일을 추적 |
| **완료 개념** | 없음 (알림만) | 있음 (done/미완료) |
| **반복** | 시간 반복 (매일 9시) | 태스크 반복 (완료 후 다음 인스턴스 생성) |
| **기한** | 없음 | 있음 (deadline) |
| **미루기** | 없음 | 있음 (postpone/skip) |
| **상태 추적** | 발송 성공/실패 | 미완료 → 완료/건너뜀/미룸 |

## 2. 사용자 시나리오

### 시나리오 1: 단발성 할일

> 사용자가 "내일까지 보고서 제출"이라는 할일을 만든다.
> 당일 오전에 상기 알림을 받고, 오후 3시에 "지금 해" 알림을 받는다.
> 완료 후 체크하면 할일이 done 처리된다.

### 시나리오 2: 반복 할일

> 사용자가 "매주 수요일 우유 사기"라는 반복 할일을 만든다.
> 매주 수요일마다 새 할일 인스턴스가 생성된다.
> 장기 출장 시 해당 주를 "건너뛰기" 처리한다.
> 수요일에 못 사면 "미루기"로 목요일로 옮긴다.

### 시나리오 3: 할일 모아보기

> 사용자가 "할일" 탭에서 오늘/이번주 할일을 한눈에 본다.
> 미완료 할일은 상단, 완료된 건 하단에 표시된다.
> 기한 초과 할일은 빨간색으로 강조된다.

### 시나리오 4: 그룹 할일

> "출장 준비"라는 그룹 아래 "항공권 예약", "호텔 예약", "짐 싸기"를 묶는다.
> 그룹 진행률(2/3 완료)이 표시된다.

## 3. 핵심 개념 설계

### 3.1 알림 3단계 (시간 기반 상태 전이)

할일의 가장 핵심적인 차별점이다. 기존의 단순 알림(시간 → 알림)이 아니라,
할일의 생명주기에 따라 **3단계로 알림 강도가 달라진다**.

```
시간 흐름 →

[상기 시점]          [알림 시점]          [기한 시점]
    │                    │                    │
    ▼                    ▼                    ▼
 ┌──────┐           ┌──────┐           ┌──────────┐
 │ 상기  │    →      │ 알림  │    →      │ 기한 초과 │
 │Remind │           │Alert │           │ Overdue  │
 └──────┘           └──────┘           └──────────┘
   가벼운              즉각적              놓침 표시
  노티피케이션         화면 알림            빨간 배지
  "잊지 마세요"       "지금 하세요"        "기한 지남!"
```

**각 단계의 차이:**

| 단계 | 이름 | 트리거 | 동작 | 예시 |
|------|------|--------|------|------|
| 1단계 | **상기 (Remind)** | 기한 N시간/일 전 | 일반 푸시 알림 | "내일 보고서 마감이에요" |
| 2단계 | **알림 (Alert)** | 설정한 알림 시각 | 강조 알림 (진동/소리) | "지금 우유 사러 가세요" |
| 3단계 | **기한초과 (Overdue)** | 기한 경과 후 | 앱 내 빨간 배지 + 미완료 목록 상단 고정 | "⚠️ 보고서 마감 지남" |

**설정 방식:**

```typescript
interface TodoTiming {
  dueDate?: string;        // YYYY-MM-DD - 기한일
  dueTime?: string;        // HH:mm - 기한 시각 (없으면 하루 종일)
  remindBefore?: number;   // 상기: 기한 N분 전 (기본: 1440 = 하루 전)
  alertTime?: string;      // 알림: 특정 시각 (HH:mm). 없으면 기한 시각 사용
}
```

> **내 의견**: 3단계 알림은 이 기능의 가장 독창적인 부분이다. 다만 사용자가 세 가지를 모두 설정하는 것은 복잡할 수 있으므로, **기본값을 잘 설정하고 고급 설정은 접어두는 UI**가 중요하다.
> - 기본 모드: 기한만 설정 → 상기(하루 전), 알림(기한 시각), 기한초과(자동) 자동 적용
> - 고급 모드: 각 단계를 개별 커스텀

### 3.2 반복 할일 (Recurring Todo)

**기존 "반복 알람"과의 핵심 차이:**

```
반복 알람 (기존):
  매일 9시 알림 → 매일 9시에 알림이 울림 (완료 개념 없음)

반복 할일 (신규):
  매주 수요일 우유 사기 → 이번 주 수요일 인스턴스 생성
                       → 완료/건너뛰기/미루기 처리
                       → 다음 주 수요일 인스턴스 자동 생성
```

**계층 구조:**

```
┌─ RecurringTodo (반복 할일 템플릿) ──────────────────┐
│  제목: "우유 사기"                                    │
│  반복: 매주 수요일                                    │
│  알림: 수요일 오전 10시                               │
│                                                      │
│  ┌─ TodoInstance (2/5주)  ──┐                        │
│  │  상태: ✅ 완료           │                        │
│  │  완료일: 2/5 11:30       │                        │
│  └──────────────────────────┘                        │
│  ┌─ TodoInstance (2/12주) ──┐                        │
│  │  상태: ⏭️ 건너뜀         │                        │
│  │  사유: 출장              │                        │
│  └──────────────────────────┘                        │
│  ┌─ TodoInstance (2/19주) ──┐                        │
│  │  상태: ⬜ 미완료         │ ← 현재               │
│  └──────────────────────────┘                        │
└──────────────────────────────────────────────────────┘
```

> **내 의견 — "반복할일"을 상위 개념으로 둘 것인가?**
>
> 유저의 질문: "할일의 상위에 반복할일이라는 개념을 두는게 나을까?"
>
> **권장: 같은 레벨, 속성으로 구분한다.**
> - 반복할일을 별도 엔티티로 만들면 데이터 모델이 복잡해지고, 기존 Memo 테이블과의 통합이 어렵다.
> - 대신 Todo 메모에 `recurrence` 속성을 추가하고, 인스턴스는 `todoInstances` 배열로 관리한다.
> - 반복 할일은 "메모 1개 + 인스턴스 N개"의 구조가 된다.
> - UI에서 반복 아이콘(🔁)으로 시각적 구분만 하면 충분하다.

### 3.3 미루기 / 건너뛰기

| 액션 | 동작 | 사용 예 |
|------|------|---------|
| **미루기 (Postpone)** | 기한을 뒤로 이동 | "오늘 못 하겠으니 내일로" |
| **건너뛰기 (Skip)** | 이번 인스턴스 패스 (반복 할일 전용) | "출장이라 이번 주는 안 함" |

```
미루기:
  기한: 2/5(수) → 미루기(+1일) → 기한: 2/6(목)
  - 할일 자체는 유지, 기한만 변경
  - 미룬 횟수 기록 (postponeCount)
  - 알림도 함께 이동

건너뛰기 (반복 할일 전용):
  2/5(수) 인스턴스 → 건너뜀 처리 → 2/12(수) 인스턴스 자동 생성
  - 해당 인스턴스는 "skipped" 상태로 기록
  - 다음 반복으로 넘어감
```

> **내 의견**: 미루기와 건너뛰기 **둘 다** 있어야 한다.
> - "미루기"만 있으면 반복 할일에서 "이번 주는 아예 패스"를 표현하기 어렵다.
> - "건너뛰기"만 있으면 단발성 할일을 하루 미루는 것이 불가능하다.
> - 미루기 시 날짜 선택 UI가 필요하고, **빠른 미루기 프리셋**(+1일, +3일, 다음주)을 제공하면 편리하다.

### 3.4 할일 상태 머신

```
                    ┌──────────┐
          생성 ──→  │  pending  │
                    └──────────┘
                         │
              ┌──────────┼──────────────┐
              ▼          ▼              ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │completed │ │postponed │ │ skipped  │
        │  (완료)   │ │ (미뤄짐)  │ │(건너뜀)  │
        └──────────┘ └──────────┘ └──────────┘
                         │
                         ▼
                    ┌──────────┐
                    │  pending  │  (새 기한으로 재생성)
                    └──────────┘
```

**기한 초과 시:**

```
pending + (현재시각 > dueDate) → overdue (기한초과)
  │
  ├─→ 완료 처리 가능 (늦었지만 함)
  ├─→ 미루기 가능 (기한 재설정)
  └─→ 건너뛰기 가능 (반복 할일)
```

> `overdue`는 별도 상태값이 아니라 **연산된 상태**로 처리한다. (pending + dueDate < now = overdue)

## 4. 데이터 모델

### 4.1 Todo 확장 필드 (Memo 인터페이스 확장)

```typescript
// 기존 Memo 인터페이스에 추가되는 할일 관련 필드

interface Memo {
  // ... 기존 필드 유지 ...
  memoType?: 'note' | 'bookmark' | 'todo';   // 기존 'task' → 'todo'로 명확화

  // === 할일 속성 (memoType === 'todo'일 때 사용) ===
  todoStatus?: TodoStatus;
  todoPriority?: TodoPriority;
  dueDate?: string;           // YYYY-MM-DD (기존 필드 활용)
  dueTime?: string;           // HH:mm

  // 알림 3단계
  todoTiming?: TodoTiming;

  // 반복 할일
  recurrence?: Recurrence;
  todoInstances?: TodoInstance[];

  // 미루기 추적
  postponeCount?: number;

  // 그룹
  todoGroupId?: string;       // 할일 그룹 소속
}

type TodoStatus = 'pending' | 'completed' | 'skipped';
type TodoPriority = 'low' | 'medium' | 'high' | 'urgent';

interface TodoTiming {
  remindBefore?: number;      // 상기: 기한 N분 전 (0=없음, 1440=하루전)
  alertTime?: string;         // 알림: HH:mm (없으면 기한시각 사용)
  alertDays?: number[];       // 알림 요일 (반복할일용)
}

interface Recurrence {
  type: 'daily' | 'weekly' | 'monthly' | 'custom';
  interval: number;           // 반복 간격 (매 N일/주/월)
  daysOfWeek?: number[];      // 요일 [0-6] (weekly용)
  dayOfMonth?: number;        // 월 중 일자 (monthly용)
  endDate?: string;           // 반복 종료일 (없으면 무한)
  endAfter?: number;          // N회 후 종료
}

interface TodoInstance {
  id: string;
  scheduledDate: string;      // YYYY-MM-DD 예정일
  status: TodoStatus;
  completedAt?: string;       // 완료 시각
  skippedAt?: string;         // 건너뛴 시각
  skipReason?: string;        // 건너뛰기 사유 (선택)
  postponedTo?: string;       // 미뤄진 날짜
  postponeCount: number;
}
```

## 5. UI/UX 설계

### 5.1 할일 표시 방식 결정

유저의 질문: "탭(하단 네비바에 추가) vs 하나의 메모처럼 표시?"

> **내 의견: 둘 다 한다. (하이브리드 접근)**
>
> 1. **메모 목록에서는 하나의 카드로 표시** — 기존 메모 흐름 유지
> 2. **하단 네비바에 "할일" 탭 추가** — 할일만 모아보는 전용 뷰
>
> 이유:
> - 할일만 빠르게 보고 싶을 때가 있고 (→ 전용 탭)
> - 메모/북마크와 섞여 있는 맥락에서 보고 싶을 때도 있다 (→ 메모 목록)
> - 기존 4탭(홈/전체메모/알림내역/설정)에 "할일" 1탭 추가는 5탭이 되어 네비바가 좁아질 수 있다
> - **대안: 홈 화면에 "오늘의 할일" 섹션 추가 + /todos 라우트를 상단 탭이나 필터로 접근**

**권장 네비게이션 구조:**

```
하단 네비바 (5탭):
[홈 🏠] [메모 📋] [할일 ✅] [알림 🔔] [설정 ⚙️]

또는 4탭 유지 + 홈에 할일 위젯:
[홈 🏠] [전체 📋] [알림 🔔] [설정 ⚙️]
          └─ 필터탭에 "할일" 추가
```

### 5.2 할일 카드 (MemoCard 내 Todo 변형)

```
┌─────────────────────────────────────────┐

│                                         │
│ 📅 2/5(수) 18:00          🔁 매주 수   │  ← 기한 + 반복 표시
│ ⏰ 상기: 하루 전 | 알림: 10:00          │  ← 알림 3단계 요약
│                                         │
│ [미루기 📌] [건너뛰기 ⏭️]              │  ← 빠른 액션
├─────────────────────────────────────────┤
│ #장보기  #생활                           │  ← 태그
└─────────────────────────────────────────┘

// 기한 초과 시 (overdue)
┌─────────────────────────────────────────┐
│ ☐  보고서 제출              ⚠️ 기한지남  │  ← 빨간 배경 + 경고 배지
│                                         │
```

## 6. 알림 시스템 통합

### 6.1 할일 알림 vs 기존 알림 구분

```
기존 시스템:
  Memo.reminders[] → 시간 기반 반복 알림
  "매일 9시에 알려줘"

할일 시스템 (신규):
  Memo.todoTiming → 할일 기한 연동 알림
  "기한 하루 전에 상기, 당일 오전에 알림, 기한 지나면 오버듀"
```

**공존 방식:**
- 할일(todo) 타입 메모에서도 기존 `reminders[]`를 사용할 수 있다 (추가 알림)
- `todoTiming`은 기한(dueDate) 기반으로 자동 계산되는 알림
- 두 시스템은 독립적으로 동작하되, 알림내역(NotificationHistory)에 함께 기록

### 6.2 알림 채널별 동작

| 채널 | 상기 (Remind) | 알림 (Alert) | 기한초과 (Overdue) |
|------|--------------|-------------|-------------------|
| **Service Worker** | 일반 푸시 | 일반 푸시 | - (앱 내 표시) |
| **Capacitor** | 일반 알림 | 진동+소리 알림 | - (앱 내 표시) |
| **FCM** | 일반 푸시 | 우선순위 높은 푸시 | - (앱 내 표시) |
| **앱 내** | - | - | 배지 + 목록 상단 고정 |

### 6.3 반복 할일의 인스턴스 생성 시점

```
옵션 A: 미리 생성 (선택)
  - 다음 N개 인스턴스를 미리 생성
  - 장점: 오프라인에서도 확인 가능
  - 단점: 데이터 증가

옵션 B: 직전 생성 (권장) ✅
  - 현재 인스턴스가 완료/건너뛰기되면 다음 인스턴스 생성
  - 앱 시작 시 누락된 인스턴스 체크 & 생성
  - 장점: 데이터 효율적
  - 단점: 오프라인 장기 사용 시 누락 가능 (앱 열면 복구)
```

## 7. 추가 고려사항

### 7.1 그룹핑에 대한 의견

유저의 질문: "그룹핑 필요?"

> **내 의견: 1차에는 간단하게, 추후 확장한다.**
>
> **Phase 1**: 기존 `tags`와 `folders`로 그룹핑을 대체
> - 태그로 "장보기", "업무", "집안일" 등 분류
> - 폴더로 큰 카테고리 분류
>
> **Phase 2**: 전용 `TodoGroup` 도입
> - 그룹 진행률 표시 (2/5 완료)
> - 그룹 단위 미루기/건너뛰기
> - 그룹 내 할일 순서 지정
>
> 이유: 그룹핑은 유용하지만, 태그/폴더와 기능이 겹칠 수 있다.
> 먼저 태그/폴더로 충분한지 검증 후 전용 그룹을 도입하는 것이 안전하다.

### 7.2 추가로 고려해야 할 사항

**1. 완료 후 되돌리기 (Undo)**
- 실수로 완료 체크했을 때 되돌리기 필요
- 토스트 메시지로 "실행 취소" 버튼 제공 (3초간)

**2. 대량 처리 (Batch Actions)**
- 여러 할일 선택 → 일괄 완료/미루기/삭제
- 특히 기한 지난 할일이 쌓였을 때 필요

**3. 위젯/홈 화면 연동**
- Android 홈 화면 위젯에 오늘의 할일 표시
- Capacitor 플러그인으로 네이티브 위젯 제공 가능 (향후)

**4. 캘린더 뷰**
- 할일을 달력 형태로 보는 뷰
- 기한별 분포 확인, 드래그로 기한 변경
- Phase 2 이후 고려

**5. 완료 통계/스트릭**
- 연속 완료 일수 (스트릭)
- 주간/월간 완료율
- 가장 많이 미룬 할일 Top 3
- 동기부여 요소

**6. 할일 ↔ 메모 전환**
- 일반 메모를 할일로 전환 (memoType 변경)
- 할일 완료 후 메모로 보관 (기록 유지)
- 기존 checklist 가 있는 메모를 할일로 마이그레이션

**7. 알림 피로도 관리**
- 상기+알림+기한초과가 동시에 여러 할일에서 오면 부담
- "방해 금지 시간대" 설정
- 알림 병합 (여러 할일을 하나의 알림으로 묶기)
- 우선순위가 높은 것만 강제 알림, 나머지는 조용한 알림

**8. 오프라인 동작**
- 오프라인에서 할일 완료/미루기 처리
- 온라인 복귀 시 Supabase 동기화
- 기존 optimistic UI 패턴 활용

**9. 자연어 입력 (향후)**
- "매주 수요일 우유 사기" → 자동 파싱하여 반복+제목 설정
- "내일 3시까지 보고서" → 기한 자동 설정
- Phase 2 이후 검토

**10. 서브태스크 vs 체크리스트**
- 기존 `checklist` 필드가 있는데, 이것과 할일의 관계 정리 필요
- 체크리스트: 할일 내부의 세부 항목 (할일 1개에 딸린 리스트)
- 서브태스크: 별도의 할일이 부모 할일에 연결 (1:N 관계)
- Phase 1에서는 기존 체크리스트를 활용하고, 서브태스크는 Phase 2에서 검토
