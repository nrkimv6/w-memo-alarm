# 알람 토글 해제 안 됨 — 수정 계획서

## Context (배경 / 문제)

"메모 수정" 다이얼로그의 **알림 설정**에서 알람 토글을 OFF로 바꾸고 저장(수정)해도
알람이 실제로 해제되지 않고 계속 울린다.

원인은 단일 버그가 아니라, 앱이 **단일 `reminder`(객체) → `reminders`(배열, 다중 알람)**
모델로 마이그레이션되면서 **알람을 실제로 등록/취소하는 하위 계층 코드들이 갱신되지 않은 것**이다.
폼/저장 계층은 정상이다 (`MemoForm.handleSubmit` 이 `enabled:false` 가 포함된 전체
`reminders` 배열을 `memosStore.update` 로 올바르게 전달하고, `ma_memos.reminders`
컬럼에도 정상 저장됨 — `memoMapper.ts:67`).

문제가 있는 곳은 알람을 OS/SW/서버에 실제로 등록·취소하는 3개 전송 계층이다.
사용자 확인 결과 **네이티브 앱과 웹/PWA 둘 다**, **기본·추가 알람 모두** 재현됨 → 세 계층 모두 수정한다.

데이터 흐름 (정상 동작 확인됨):
`ReminderCard 토글` → `ReminderSettings.updateReminder` → `MemoForm.reminders($state)`
→ `MemoForm.handleSubmit` → `memosStore.update(id, { reminders })` → `_doUpdate`
→ (여기서부터 계층별 버그 발생).

## 근본 원인 (3개 결함)

### 결함 A — 네이티브(Capacitor) 편집 경로가 구형 단일 `reminder` 필드를 검사
**파일:** `src/lib/stores/memos.svelte.ts:547-553`
```ts
if (await isNative()) {
  if (optimisticUpdate.reminder?.enabled) {   // ← 항상 undefined
    scheduleNotification(optimisticUpdate);
  } else {
    cancelNotification(id);
  }
}
```
마이그레이션 후 `memo.reminder` 는 `undefined` (`reminderHelpers.migrateToMultipleReminders`
가 `reminder: undefined` 로 비움). 따라서 다중 알람 메모를 편집하면 **항상 `cancelNotification`
경로로 빠지고, 개별 알람의 `enabled` 상태(`reminders[]`)를 전혀 반영하지 못한다.**
재스케줄도, 부분 해제도 일관되게 동작하지 않는다.
참고: 정작 `src/lib/utils/capacitor.ts` 의 `scheduleNotification`/`cancelNotification`/
`getActiveReminders`(`reminders.filter(r=>r.enabled)`)는 배열을 올바르게 처리하므로,
**호출부만 배열 기준으로 고치면 된다.**

### 결함 B — Service Worker 가 `memoId` 기준으로만 reminder를 교체 → 형제 알람 덮어씀
**파일:** `src/service-worker.ts:549-562` (UPDATE_MEMO_REMINDER 핸들러)
```ts
scheduledReminders = scheduledReminders.filter((r) => r.memoId !== reminder.memoId);
if (reminder.time) scheduledReminders.push(reminder);
```
**파일:** `src/lib/stores/notifications.svelte.ts:660-679` (`updateReminderInServiceWorker`)
는 활성 알람마다 **UPDATE 메시지를 개별로 N번** 보낸다.
SW 핸들러는 매 메시지마다 해당 memoId의 reminder를 **전부 제거 후 1개만 push** 하므로,
다중 알람 메모는 **마지막 알람 1개만 SW에 남고 나머지는 조용히 사라진다.**
→ SW 상태가 서버/실제 설정과 불일치, 일부 알람 누락·잘못된 발화의 원인.

### 결함 C — SW로 보내는 reminder payload에 `enabled` 누락 (방어 부족)
**파일:** `notifications.svelte.ts:588-599`, `663-674` / `service-worker.ts:15-27`(인터페이스),
`402-458`(`checkScheduledReminders`)
SW의 발화 체크는 `enabled` 를 전혀 보지 않고 "비활성 알람은 애초에 등록되지 않는다"는
가정에만 의존한다. 클라이언트 필터가 한 번이라도 누락되면 비활성 알람이 그대로 울린다.

### (확인됨, 수정 불필요) 서버 푸시 경로
`src/lib/services/alarmSchedules.ts` 의 `syncMemoAlarms` 는
`deleteMemoAlarms` 후 `reminders.filter(r=>r.enabled)` 만 재생성하므로 정상.
비활성 알람은 `alarm_schedules` 에서 삭제됨 → 서버 푸시 경로는 올바름.

## 수정 계획

### 1) 네이티브 편집 경로 — `memos.svelte.ts:547-553`
`reminder` 단일 필드 대신 `reminders[]` 의 활성 알람 기준으로 분기.
`getRemindersFromMemo` 는 이미 같은 파일에서 import 되어 있음(`:648`).
```ts
if (await isNative()) {
  const activeReminders = getRemindersFromMemo(optimisticUpdate).filter(r => r.enabled);
  if (activeReminders.length > 0) {
    scheduleNotification(optimisticUpdate); // 내부에서 cancel 후 활성 알람만 재스케줄
  } else {
    cancelNotification(id);                 // 활성 알람 0개 → 전부 취소
  }
}
```
부수 확인: `capacitor.ts:scheduleNotification` 은 `activeReminders.length===0` 일 때
`cancelNotification` **이전에** early-return 한다(`:80` vs `:86`). 위처럼 호출부에서
활성>0 일 때만 호출하므로 문제 없음. (원하면 `scheduleNotification` 내부에서 cancel을
early-return 앞으로 옮기는 견고화도 가능 — 선택사항.)
`add()` 등 생성 경로도 동일 패턴을 쓰는지 점검하여 같은 기준으로 통일.

### 2) Service Worker 다중 알람 원자적 교체 — `notifications.svelte.ts` + `service-worker.ts`
메모 단위로 **활성 알람 배열을 한 번에** 보내고, SW가 해당 memoId 항목을 통째로 교체한다.

- `notifications.svelte.ts` `updateReminderInServiceWorker`: 활성 알람들을 plain object
  배열로 만들어 **단일 메시지**로 전송(빈 배열이면 제거 의미). 각 객체에 `enabled: r.enabled`
  포함(결함 C 대응). 메시지 형태:
  `{ type: SW_MSG.UPDATE_MEMO_REMINDER, memoId, reminders: [...] }`
- `service-worker.ts` UPDATE 핸들러: 
  ```ts
  const { memoId } = event.data;
  const incoming = (event.data.reminders ?? []).filter((r) => r.time);
  scheduledReminders = scheduledReminders.filter((r) => r.memoId !== memoId);
  scheduledReminders.push(...incoming);
  ```
  → "일부 해제 / 다중 알람 / 전부 해제" 모두 한 번에 정확히 반영.
- 하위호환: SW 롤아웃 중 구형 메시지(`reminder` 단일)도 수용하도록 핸들러에서
  `event.data.reminders ?? (event.data.reminder ? [event.data.reminder] : [])` 형태로 방어.
- `REGISTER_MEMO_REMINDERS`/`REMOVE_MEMO_REMINDER`(memoId 기준)는 그대로 유지.

### 3) `enabled` 플래그 전파 + SW 발화 체크 (방어, 결함 C)
- `notifications.svelte.ts` 의 `registerRemindersToServiceWorker`(`:588-599`)와
  `updateReminderInServiceWorker` 가 보내는 reminder 객체에 `enabled: r.enabled` 추가.
- `service-worker.ts` `ScheduledReminder` 인터페이스(`:15-27`)에 `enabled?: boolean` 추가.
- `checkScheduledReminders`(`:415` 루프 시작부)에 `if (reminder.enabled === false) return;` 가드 추가.

> 핵심 수정은 1)·2). 3)은 한 계층 필터가 빠져도 비활성 알람이 울리지 않게 하는 안전망.
